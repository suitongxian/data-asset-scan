在日常盘点模式下，classifySurvivalStatus 方法（第 861-890 行）将文件分为三类：
1. 新文件：之前没有，现在有 → 会计算哈希并插入
2. 正常文件：之前有，现在也有 → 只增加 scan_found_count，没有重新计算哈希
3. 删除文件：之前有，现在没有 → 标记为删除

问题出在第 978-995 行处理"正常文件"的逻辑：

// 4. 处理正常文件：批量更新 scan_found_count + 1
if (classification.normalFileIds.length > 0) {
this.dataRepo.batchIncrementScanFoundCount(classification.normalFileIds)
scannedCount += classification.normalFileIds.length
// ...
}

当文件被修改后，其路径仍然存在，所以被归类为"正常文件"，但代码只是简单地增加 scan_found_count，而没有检查文件内容是否发生变化并更新 content_sign。

修复方案：

需要在处理"正常文件"时，比对文件的file_update_time（修改时间）与数据库中记录的时间，如果发生变化，则重新计算哈希并更新content_sign
